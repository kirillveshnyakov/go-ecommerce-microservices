# Go E-Commerce Microservices

Go E-Commerce Microservices - учебный микросервисный проект на Go, реализующий базовую платформу электронной коммерции: каталог товаров, корзину пользователя, управление заказами, учет остатков и асинхронную доставку уведомлений о статусах заказов.

Дополнительно в проект включен frontend на React/Vite, который предоставляет пользовательский интерфейс поверх backend части приложения.

Ключевой акцент сделан на микросервисной декомпозиции, gRPC-взаимодействии, надежной обработке событий через outbox/inbox-подход, работе с PostgreSQL, генерации API-контрактов из protobuf и тестировании всей системы.

## Возможности

**Cart - корзина пользователя**

- ведение пользовательской корзины как отдельной части покупательского сценария;
- расчет итоговой стоимости на основе актуальных данных о товарах;
- оформление корзины в заказ без владения каталогом и складской логикой.

**LOMS - логистика и управления заказами**

- управление каталогом, остатками и заказами в одном доменном сервисе;
- резервирование и освобождение товаров в рамках жизненного цикла заказа;
- фиксация изменений статуса заказа как событий для асинхронной обработки.

**Notifications - асинхронные уведомления**

- асинхронная доставка уведомлений о смене статуса заказа через Kafka;
- outbox/inbox-подход для надежной обработки событий;
- повторная доставка callback при ошибках с at-least-once семантикой.

**API и пользовательский интерфейс**

- gRPC API для межсервисного взаимодействия;
- HTTP gateway поверх protobuf-контрактов для внешних клиентов;
- frontend на React/Vite для сценариев каталога, корзины и заказа;
- OpenAPI-спецификации, генерируемые из protobuf-контрактов.

**Качество и эксплуатация**

- миграции базы данных для каждого сервиса;
- SQL-запросы и типобезопасный доступ к базе через sqlc;
- unit-тесты для use case, repository, controller и adapter слоев;
- интеграционные тесты для проверки полного сценария через Docker Compose;
- линтинг через golangci-lint;

## Архитектура

Проект организован как monorepo с несколькими независимыми Go-сервисами. Сервисы разделены по бизнес-ответственности.

Синхронные операции выполняются через gRPC и HTTP gateway, а события изменения статуса заказа доставляются асинхронно через Kafka. Для надежности используются outbox в `loms` и inbox в `notifications`.

| Компонент | Назначение |
| --- | --- |
| `cart` | Пользовательская корзина и точка входа в оформление заказа. Собирает позиции, считает итоговую стоимость и передает checkout в `loms`. |
| `loms` | Центр управления заказом: каталог, остатки, резервирование и статусы. Отвечает за консистентное изменение заказа и генерацию событий. |
| `notifications` | Асинхронная доставка уведомлений. Работает с другими сервисами через Kafka и gRPC. |
| `pkg` | Общие компоненты и сгенерированные protobuf-контракты. |
| `frontend` | React/Vite интерфейс для каталога, корзины и заказов. |
| `integration-tests` | End-to-end проверки полного окружения: сервисы, PostgreSQL, Kafka, gRPC и gateway. |

PostgreSQL используется как основное хранилище данных сервисов. Kafka отвечает только за событийную часть: доставку уведомлений о смене статусов заказов.

## API

Сервисы описаны через protobuf-контракты и доступны по gRPC. Для внешних клиентов также используется HTTP gateway.

| Сервис | HTTP endpoint | Назначение |
| --- | --- | --- |
| `cart` | `POST /v1/cart/item/add` | Добавить товар в корзину |
| `cart` | `POST /v1/cart/item/delete` | Удалить товар из корзины |
| `cart` | `POST /v1/cart/list` | Получить содержимое корзины |
| `cart` | `POST /v1/cart/clear` | Очистить корзину |
| `cart` | `POST /v1/cart/checkout` | Оформить корзину в заказ |
| `loms` | `POST /v1/product/create` | Создать товар |
| `loms` | `POST /v1/product/info` | Получить информацию о товаре |
| `loms` | `POST /v1/stock/info` | Получить остаток товара |
| `loms` | `POST /v1/stock/set` | Изменить остаток товара |
| `loms` | `POST /v1/order/create` | Создать заказ |
| `loms` | `POST /v1/order/info` | Получить информацию о заказе |
| `loms` | `POST /v1/order/pay` | Оплатить заказ |
| `loms` | `POST /v1/order/cancel` | Отменить заказ |
| `notifications` | `POST /v1/notifications/order/status-changed` | Отправить уведомление о смене статуса заказа |

## Уведомления

Для доставки уведомлений используется связка outbox, Kafka и inbox. Этот поток спроектирован под at-least-once семантику: событие может быть доставлено повторно, но не должно потеряться из-за временной ошибки callback-сервиса или сбоя обработчика.

В `loms` событие изменения статуса заказа сначала сохраняется в PostgreSQL вместе с бизнес-изменением. После этого отдельный worker читает outbox и публикует событие в Kafka. Это убирает отправку уведомления из критичного пути изменения заказа.

В `notifications` входящее событие сохраняется в inbox. Доставка callback выполняется отдельно, с учетом повторных попыток. Событие помечается обработанным после успешной отправки. Если событие не удалось доставить за заданное число попыток, оно остается для ручной обработки.

Событие уведомления содержит идентификатор пользователя, идентификатор заказа и новый статус:

```json
{
  "user_id": 900001,
  "order_id": 123,
  "status": "awaiting_payment"
}
```

## Требования

- Go 1.26.2
- Docker и Docker Compose
- Node.js и npm для frontend
- go-task

На Windows рекомендуется запускать команды через WSL или Git Bash: часть команд в `Taskfile.yml` использует bash-синтаксис.

## Быстрый старт

Установить go-task:

```bash
go install github.com/go-task/task/v3/cmd/task@latest
```

Сгенерировать protobuf/gRPC/gateway-код и mock файлы для тестов:

```bash
task generate
```

Запустить backend-инфраструктуру:

```bash
task backend
```

Команда поднимает Go-сервисы, PostgreSQL, Kafka и Kafdrop через Docker Compose.

Запустить frontend:

```bash
task frontend
```

Frontend будет доступен на:

```text
http://localhost:5173
```

Полезные task-команды:

| Команда | Назначение |
| --- | --- |
| `task backend` | Запустить backend-окружение через Docker Compose |
| `task frontend` | Установить frontend-зависимости и запустить Vite dev server |
| `task test` | Запустить интеграционные тесты в Docker Compose |
| `task lint` | Запустить golangci-lint по Go-сервисам |
| `task generate` | Установить инструменты генерации и сгенерировать protobuf/gRPC/gateway-код |
| `task fast-generate` | Повторить генерацию без переустановки инструментов |
| `task down` | Остановить окружение и удалить volumes |

## Конфигурация

Сервисы конфигурируются через переменные окружения. 

Общие переменные для сервисов, работающих с PostgreSQL:

| Переменная | Назначение |
| --- | --- |
| `POSTGRES_HOST` | Адрес PostgreSQL |
| `POSTGRES_PORT` | Порт PostgreSQL |
| `POSTGRES_DB` | Имя базы данных |
| `POSTGRES_USER` | Пользователь базы данных |
| `POSTGRES_PASSWORD` | Пароль пользователя |

Основные переменные сервисов:

| Переменная | Сервис | Назначение |
| --- | --- | --- |
| `GRPC_PORT` | `cart`, `loms`, `notifications` | Порт gRPC-сервера |
| `GRPC_GATEWAY_PORT` | `cart`, `loms` | Порт HTTP gateway |
| `GRPC_SHUTDOWN_TIME` | `cart`, `loms`, `notifications` | Таймаут graceful shutdown для gRPC-сервера |
| `HTTP_SHUTDOWN_TIME` | `cart`, `loms` | Таймаут graceful shutdown для HTTP gateway |
| `LOMS_GRPC_ADDR` | `cart` | gRPC-адрес сервиса `loms` |
| `NOTIFICATIONS_GRPC_ADDR` | `loms` | gRPC-адрес сервиса `notifications`, если используется синхронный transport |
| `KAFKA_BROKERS` | `loms`, `notifications` | Адреса Kafka brokers |
| `KAFKA_NOTIFICATIONS_TOPIC` | `loms`, `notifications` | Topic для событий статуса заказа |
| `KAFKA_CONSUMER_GROUP` | `notifications` | Consumer group сервиса уведомлений |
| `CALLBACK_ADDR` | `notifications` | Адрес внешнего callback-сервиса |
| `TELEGRAM_TOKEN` | `notifications` | Токен Telegram bot для альтернативной доставки уведомлений |
| `TELEGRAM_CHAT_ID` | `notifications` | Chat ID получателя Telegram-уведомлений |

Outbox в `loms`:

| Переменная | Назначение |
| --- | --- |
| `OUTBOX_WORKERS` | Количество outbox workers |
| `OUTBOX_BATCH_SIZE` | Размер пачки событий для обработки |
| `OUTBOX_FETCH_PERIOD` | Период опроса outbox |
| `OUTBOX_IN_PROGRESS_TTL` | TTL для событий, находящихся в обработке |

Inbox в `notifications`:

| Переменная | Назначение |
| --- | --- |
| `INBOX_WORKERS` | Количество inbox workers |
| `INBOX_MAX_ATTEMPTS` | Максимальное число попыток доставки |
| `INBOX_BATCH_SIZE` | Размер пачки событий для обработки |
| `INBOX_FETCH_PERIOD` | Период опроса inbox |
| `INBOX_RETRY_DELAY` | Задержка перед повторной попыткой доставки |
| `INBOX_IN_PROGRESS_TTL` | TTL для событий, находящихся в обработке |

## Тестирование

Проект покрыт unit- и интеграционными тестами.

Запустить unit-тесты Go-сервисов:

```bash
go test ./cart/...
go test ./loms/...
go test ./notifications/...
```

Запустить полный интеграционный сценарий:

```bash
task test
```

Запустить линтер:

```bash
task lint
```

Unit-тесты проверяют отдельные слои сервисов:

- бизнес-логику корзины, товаров, остатков и заказов;
- конвертеры между transport- и domain-моделями;
- repository-слой;
- gRPC-контроллеры;
- Kafka producer и consumer;
- outbox и inbox workers.

Интеграционные тесты поднимают окружение с несколькими экземплярами сервисов, PostgreSQL и Kafka. Они проверяют полный путь запроса через gRPC и HTTP gateway, корректность работы базы данных, создание заказов, изменение статусов и доставку событий уведомлений через Kafka.
